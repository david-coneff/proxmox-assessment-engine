### R3-001 · HIGH · Phase-08 FORGE_INCOMPLETE banner still blocked by exit 1

**File:** `proxmox-bootstrap/forge_scripts.py:729–738`

**Root cause:** When phases 04/05/06 exit 2 (NOT_IMPLEMENTED), forge.sh correctly sets `_forge_incomplete=1` and continues. Phase-07 succeeds (exit 0). Phase-08 then runs and executes:

```bash
kubectl wait node --all --for=condition=Ready --timeout=120s || {
  echo "[phase-08] k3s nodes not ready after 120s." >&2
  checkpoint_failed "$step"
}
```

On a system where k3s has not been provisioned (the standard NOT_IMPLEMENTED path), `kubectl` fails because no k3s cluster exists. `checkpoint_failed "$step"` calls `exit 1`. The forge.sh loop sees `_phase_exit=1` and takes the fatal path:

```bash
echo "[forge] FAIL phase-08-verify.sh (exit 1) — aborting." >&2
exit 1
```

The FORGE_INCOMPLETE banner at `forge_scripts.py:244–265` is never reached. This is the same structural pattern as N-002 (phase-06 previously exiting 1 instead of 2).

**Correct fix:** When `kubectl wait node` fails and no k3s cluster is present (a precondition-not-met condition, not a k3s-is-broken condition), phase-08 should exit 2 (NOT_IMPLEMENTED), not 1. The distinction: "k3s nodes not ready because k3s was never installed" → exit 2; "k3s is installed but nodes are not Ready after 120 seconds" → exit 1 (fatal, cluster is broken).

**Detection heuristic:** Check whether `/etc/rancher/k3s/k3s.yaml` or `k3s` binary exists before running `kubectl wait`. If neither is present, k3s was never installed → exit 2.

### R3-002 · LOW · Headscale CLI command mismatch in spawn-planner.py and federated_reconstruction.py

**Files:**  
- `proxmox-bootstrap/spawn-planner.py:129` (`_generate_headscale_auth_key`)  
- `proxmox-bootstrap/federated_reconstruction.py:394`

`spawn-planner.py` (interactive CLI) uses the old headscale subcommand format:

```python
["headscale", "authkeys", "generate",
 "--expiration", "1h",
 "--user", "broodforge"]
```

The current headscale CLI (≥0.17) uses `preauthkeys create`, not `authkeys generate`. The canonical format used in `spawn_planner.py` (library), `NODE-SPAWNING.md`, and spawn error messages is:

```
headscale preauthkeys create --expiration 1h --user broodforge
```

`federated_reconstruction.py:394` generates step-by-step instructions with the old format:

```python
f"3. On {coordinator_cell}: headscale authkeys generate --expiration 1h --user broodforge",
```

**Impact:** Low — `spawn-planner.py` has a graceful fallback to a manual prompt when `subprocess.run` fails (`FileNotFoundError` or non-zero exit). The old command will fail silently and the operator will be prompted. `federated_reconstruction.py` is a documentation generator; incorrect instructions are confusing but not a runtime blocker.

### R3-004 · HIGH (latent) · KEEPASS_MASTER_PASSWORD not propagated to phases 05–06 subprocesses

**File:** `proxmox-bootstrap/forge_scripts.py:399, 579–638, 641–688`

**Architecture:** Each forge phase runs as a separate `bash` subprocess invoked by forge.sh:

```bash
bash "$SCRIPT_DIR/$script"
```

All phase scripts source `lib/forge-keepass-gate.sh` via `_phase_header`. `forge_keepass_gate()` is available in all phases but is only *called* in phase-03 (`forge_scripts.py:399`).

`KEEPASS_MASTER_PASSWORD` is deliberately not exported (security design — prevents child process inheritance). `FORGE_KDBX_UNLOCKED` is also not exported. When phase-05 or phase-06 starts as a fresh bash subprocess, both variables are unset. `kdbx_get()` pipes an empty string to `keepassxc-cli`, which returns empty output for every key.

**Current impact (latent):** Phase-05's `kdbx_get` calls are inside `if [ -f "$_inv" ] && [ -f "$_play" ]; then` (the ansible-exists branch). In the NOT_IMPLEMENTED path (no ansible inventory), the branch is skipped and phase-05 exits 2 before any `kdbx_get` is called. Phase-06's `kdbx_get` returns empty → `FORGEJO_TOKEN` is empty → exit 2 (NOT_IMPLEMENTED). The current exit-2 path happens to produce the correct observable behavior, masking the underlying defect.

**Future impact (active bug):** When an operator completes manual VM provisioning and adds Forgejo credentials to KeePass, phase-06 will still exit NOT_IMPLEMENTED even though credentials are present — `kdbx_get` always returns empty because `forge_keepass_gate` was never called in this subprocess. The FORGE_INCOMPLETE banner will instruct the operator to "add credentials to KeePass" when they already have. This creates an unresolvable loop: the operator cannot progress by following the banner's instructions.

**Correct fix:** Each phase that calls `kdbx_get` must call `forge_keepass_gate` first. To prevent re-prompting the operator once per phase, `forge_keepass_gate` needs cross-process idempotency: after the first unlock (phase-03), persist the password to a 0600 tmpfs session file (e.g., `/run/broodforge-forge.session`). Subsequent phases check for and read the session file instead of prompting. forge.sh cleans up the session file on completion or abort.
