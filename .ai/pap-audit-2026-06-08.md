**Audit date:** 2026-06-08  
**Auditor:** Claude (PAP-AUDIT analytical mode)  
**Audit type:** Comprehensive code + workflow audit  
**Prior audit:** `pap/audits/2026-06-07_broodforge-pap-audit.md` (governance/charter only — narrow scope, not duplicated here)  
**Scope:**
- Incomplete implementations (stubs, TODOs, phases that exit-0 unconditionally)

## Executive Summary

The codebase is well-structured and internally consistent at the module level. The Phase 1.H–1.K implementations are solid and meet their AD goals. The primary risk concentration is in the **forge workflow**: the most consequential phases (04 — VM provisioning, 05 — k3s cluster setup) both self-mark as successful when their work is absent, producing a "forge complete" outcome on a machine with no VMs and no k3s cluster. This is documented honestly in FORGING.md but the operational framing (`bash forge.sh` → "Your hatchery is operational") creates a material expectation gap.

A secondary risk is **dashboard assessment scores**: every deployed dashboard instance renders the "No readiness report found" placeholder because the score keys it reads (`ACS/RRS/DCS/CRS/OSS/PHS`) are never populated by any module in the codebase.

All other findings are bounded and individually addressable.

## Part 1 — Incomplete Implementations

### F-001 · CRITICAL · Forge phase-04 (VM provisioning) self-succeeds with no work done

**File:** `proxmox-bootstrap/forge_scripts.py:492–505`

`generate_phase_04_sh()` checks for an `opentofu/` directory and, finding none (the forge package currently contains no such directory), prints advisory messages and calls `checkpoint_done "$step"` unconditionally. The forge orchestrator records phase 04 as complete and continues to phase 05. No VMs are provisioned.

```bash
# generate_phase_04_sh (line 499-504):
if [ -d "$SCRIPT_DIR/opentofu" ]; then
    tofu apply ...
else
    echo "[phase-04] No opentofu/ modules in package — VM provisioning skipped." >&2
    echo "[phase-04] Provision the Forgejo + operations VMs manually ..." >&2
fi
checkpoint_done "$step"   # ← unconditional: marks success whether or not VMs exist
```

`forge.sh` then prints "Forge complete. Your hatchery is operational." The hatchery has no VMs and is not operational.

FORGING.md status table documents this honestly ("⚠️ Manual for now") but the table appears well below the main runbook narrative. An operator following "bash forge.sh" without reading the status table will get a misleading success signal.

**Impact:** An operator following the primary runbook path will believe forging succeeded when it has not. All subsequent spawn and GitOps operations depend on VMs that do not exist.

### F-002 · CRITICAL · Forge phase-05 (k3s) self-succeeds with no work done

**File:** `proxmox-bootstrap/forge_scripts.py:515–536`

`generate_phase_05_sh()` checks for `ansible/inventory/hosts.yaml` and the k3s playbook. The forge package currently generates neither file. Finding them absent, the script prints a warning and calls `checkpoint_done "$step"`. k3s is never installed.

Same unconditional `checkpoint_done` pattern as F-001. Same misleading success outcome.

**Impact:** Even if phase-04 were fixed (VMs manually provisioned), phase-05 would still claim success without a k3s cluster existing. All Flux CD, GitOps, and broodling join operations depend on a working k3s cluster.

## Part 2 — Conceptual Gaps

### F-008 · HIGH · Dashboard assessment scores always render as "No readiness report found"

**File:** `proxmox-bootstrap/broodforge_dashboard.py:312–319`  
**Cross-ref:** `proxmox-bootstrap/_recovery_readiness_certificate.py:27–56` (documents the discrepancy)

`_scores_from_readiness()` reads keys `ACS`, `RRS`, `DCS`, `CRS`, `OSS`, `PHS` from the readiness report dict. No module in the codebase populates these keys. `readiness.py::score_graph()` produces `overall_score` / `overall_score_reason` / `components` — not the five named abbreviations.

`_recovery_readiness_certificate.py` documents this explicitly: "The five-letter abbreviation scheme appears only as aspirational/defensive UI code in broodforge_dashboard.py, reading a 'scores' dict nothing populates."

Result: `scores_html` is always an empty string, and the dashboard falls through to:
```python
if not scores_html:
    scores_html = '<p ...>No readiness report found — run engine.py ...</p>'
```
This message is shown on every deployed dashboard instance even when a Readiness-Report.json exists and has valid data.

**Impact:** The primary visual section of the dashboard ("Assessment Scores") is permanently non-functional. Operators cannot use the dashboard to assess cell health via the readiness signal.

### F-009 · HIGH · bootstrap-state.json path mismatch between forge scripts and dashboard default

**Files:**  
- `proxmox-bootstrap/forge_scripts.py:582` (phase-07 writes to `$SCRIPT_DIR/proxmox-bootstrap/bootstrap-state.json`)
- `proxmox-bootstrap/broodforge_dashboard.py:74` (`DEFAULT_STATE = "/var/lib/broodforge/bootstrap-state.json"`)

Phase-07 of forge.sh writes bootstrap-state.json into `$SCRIPT_DIR/proxmox-bootstrap/bootstrap-state.json` (relative to the unpacked forge package, typically `/root/proxmox-bootstrap/bootstrap-state.json`). The dashboard's default state path is `/var/lib/broodforge/bootstrap-state.json`. Unless the operator explicitly configures the `--state` flag or edits `dashboard.json`, the dashboard starts up reading a different (or non-existent) state file and shows empty data.

### F-010 · HIGH · Dashboard POST endpoints are unprotected by default (action_token empty)

**File:** `proxmox-bootstrap/broodforge_dashboard.py:916–925`

```python
def _check_action_token(self) -> bool:
    token = self.headers.get("X-Broodforge-Token", "")
    expected = self._cfg.action_token
    if not expected:
        return True   # no token configured; allow (dev mode)
```

`DashboardConfig.action_token` defaults to `""`. An `ensure_token()` call at startup generates and saves a token only if none is set. If `dashboard.json` doesn't exist yet when the server starts for the first time, a token IS generated and saved. However, if the operator starts the dashboard with `--config /some/nonexistent/path` or if the auto-generated token is somehow cleared, all POST actions (remediation approve/reject, batch-approve, analyze-failures) require no authentication.

More concretely: `ensure_token()` calls `self.save()` to persist the token, but `save()` can fail silently if the config directory is not writable (it calls `os.makedirs` with no error handling for permission denied). If `save()` fails, `self.action_token` is set in memory but not persisted; on next restart, `action_token` is again empty and all POSTs are unprotected.

### F-011 · HIGH · k3s join tokens are `None` in spawn-plan.json on freshly-forged systems

**File:** `proxmox-bootstrap/spawn_planner.py:468–470`

```python
_k3s_secrets = bootstrap_state.get("k3s") or {}
...
"worker_join_token":  _k3s_secrets.get("worker_join_token"),
"server_join_token":  _k3s_secrets.get("server_join_token"),
```

Phase-07 of forge.sh seeds bootstrap-state.json from forge-manifest.json via `init-bootstrap-state.py`. `forge-manifest.json` (produced by `forge_planner.py`) does not contain k3s join tokens — it contains a `keepass_config` block with a `k3s_token_path` reference, not actual token values.

Actual k3s tokens are created during phase-05 (Ansible k3s-server playbook) and stored in KeePass. Nothing in phases 05–08 extracts those tokens from KeePass and writes them back into bootstrap-state.json's `k3s` section.

Result: `bootstrap_state["k3s"]` is either absent or empty, so both `worker_join_token` and `server_join_token` in generated spawn plans are `None`. The generated ansible k3s vars file will have `null` tokens, and spawn will fail at the k3s join step on every broodling.

### F-014 · MEDIUM · Dashboard's doc-file serving fails in production (path traversal upward to find `docs/`)

### F-016 · LOW · `_read_latest_drill()` in `_recovery_readiness_certificate.py` reads `drills[0]` not latest

**File:** `proxmox-bootstrap/_recovery_readiness_certificate.py:327–333`

```python
drills = state.get("reconstruction_drills") or []
if drills:
    return drills[0]
```

`drills[0]` is the first drill in the list (oldest), not `drills[-1]` (newest). If `reconstruction_drills` is appended chronologically (oldest first), this returns the oldest drill record as the "latest" for the certificate. If the list is prepended (newest first), it is correct. No comment or convention establishes which ordering is authoritative. The certificate's `latest_drill` section will silently contain stale data on any system with more than one drill.

## Part 3 — User Workflow Logic Gaps

### F-017 · CRITICAL · The forge "complete" message is issued even when the hatchery is not operational

**Workflow:** Forge → end-to-end

Cross-references F-001 and F-002. The compounded effect: forge.sh runs all 9 phases, checkpoints every phase as complete (including phases 04 and 05 which self-succeed with no work), and prints:

```
=================================================================
 Forge complete. Your hatchery is operational.
 Hostname:  pve01
 Cell ID:   pve01-cell
 Next steps:
   1. Push your GitOps repository to Forgejo
   2. Flux CD will deploy application workloads
   ...
=================================================================
```

The next-steps suggest pushing to Forgejo and deploying Flux workloads. Forgejo does not exist (no VMs). There is no k3s cluster to receive Flux workloads. The operator is directed toward impossible next steps with no clear signal that the critical middle section of forging was skipped.

**Gap type:** Missing prerequisite enforcement + false success signal.

### F-018 · HIGH · answer.toml in image builder contains two mandatory placeholders operators must manually resolve

**File:** `proxmox-bootstrap/_image_builder.py:207, 180`

The generated `answer.toml` includes two values the operator MUST manually replace before the automated installer can work:
1. `filter.ID_NET_NAME = "__POPULATE_INTERFACE_NAME__"` — the network interface name must be discovered from the target hardware (e.g., `enp3s0`) and manually edited.
2. `disk-list = ["__POPULATE_DISK_DEVICE__"]` (when `disk_list` is not provided at build time) — the disk device path must be discovered and edited.

The README inside the bundle documents this requirement, but the CLI (`generate-bootstrap-image.py`) has no flag that forces the operator to supply disk/interface values before building. An operator who runs `generate-bootstrap-image.py --manifest forge-manifest.json` gets a bundle with unfilled placeholders and an installer that will fail at either network or disk setup with cryptic errors — not a clear "you must fill in X before using this."

**Gap type:** Missing prerequisite enforcement, assumed knowledge.

### F-019 · HIGH · Forge phase-06 (Flux CD) depends on Forgejo admin password in KeePass, but KeePass was initialized with placeholders

**Workflow:** Forge phase-03 → phase-06

Phase-03 initialises the KeePass database and creates entries with "PLACEHOLDER — set during service deploy" notes. `render_init_commands()` in `forge_keepass_init.py` creates every entry with `--no-password` (no stored value). Phase-06 then does:

```bash
FORGEJO_TOKEN="$(kdbx_get 'Infrastructure/forgejo/admin-password')"
flux bootstrap gitea --token-auth --personal ...
```

`kdbx_get` will either return empty string (if `keepassxc-cli show` returns nothing for an empty-valued entry) or "MANUAL_ENTRY_REQUIRED" (if keepassxc-cli is absent). Either way, `flux bootstrap` receives an invalid token and fails.

The FORGING.md troubleshooting section acknowledges "Forgejo credentials aren't in KeePass yet" as a cause of phase-06 failure, but doesn't document the workflow for populating KeePass credentials between phase-03 and phase-06. An operator must understand that phase-04 (manual VM provisioning), phase-05 (manual k3s setup), and Forgejo admin account creation must all happen before resuming at phase-06 — and that KeePass must be updated with real credentials before phase-06 can work.

**Gap type:** Catch-22 (phase-06 depends on credentials that only exist after the manual steps that replace phases 04-05).

### F-020 · HIGH · Spawn plan embeds `None` for k3s join tokens; broodling k3s join will fail

**Workflow:** Spawn → broodling k3s join

Cross-references F-011. The spawn package's generated Ansible k3s vars file (from `generate_ansible_k3s_vars()` in `spawn_iac_generator.py`) will contain `k3s_token: null` or an empty string. The Ansible k3s-worker role will either fail immediately (if it validates the token) or attempt to join with an empty token (which k3s rejects with "cluster init failed: could not fetch auth token").

The spawn workflow's own comment notes the problem: "k3s join tokens are the one secret the spawn package carries (README: included in the spawn package, valid only for the spawn window, rotated after the broodling joins). Carry them so generate_ansible_k3s_vars() emits a usable token instead of the unresolved `{{ vault_k3s_worker_token }}` placeholder, which nothing fills." But the fix (populating bootstrap-state.json's `k3s` section with real tokens) is not implemented.

**Gap type:** Orphaned output — forge populates KeePass but never writes tokens into bootstrap-state.json; spawn reads bootstrap-state.json, not KeePass.

### F-027 · LOW · `forge_keepass_init.py` TOTP provisioning fires BEFORE the operator has verified KeePass init succeeded

**Workflow:** Forge phase-03 → KeePass init → TOTP setup

The execution order in `forge_keepass_init.py::__main__` is:

## Part 4 — Cross-Cutting Concerns

### F-028 · CRITICAL · Bare-hardware-to-running-cell path has a documented but operationally non-obvious gap

**Affected docs:** `proxmox-bootstrap/FORGING.md`, `forge_scripts.py`

The complete path from bare hardware to a running hatchery requires:
1. Steps 1–3 (manifest, package, copy) — fully automated
2. Phase 00–03 — automated (hardware discovery, planning, validation, host config)
3. **Phase 04 — manual VM provisioning** (no opentofu/ modules; operator must create VMs by hand or bring their own IaC)
4. **Phase 05 — manual k3s setup** (no ansible inventory; operator must configure k3s manually or bring their own inventory)
5. Phase 06 — scripted (Flux bootstrap, but requires real Forgejo credentials from step 4)
6. Phases 07–08 — automated

The FORGING.md status table (well-buried in Step 5) accurately describes this, but the high-level runbook narrative ("run bash forge.sh — it runs 9 phases in order") obscures it. The path is not a continuous `bash forge.sh` — it is a `bash forge.sh`, then manual VM/k3s work, then `bash forge.sh --from 6`.

No document shows the full correct sequence prominently. The critical instructions are in two separate places (status table in Step 5, resume command at the bottom of the table).

### F-032 · MEDIUM · `_check_action_token()` allows all POSTs if `dashboard.json` config write fails silently

**Cross-reference:** F-010 (full analysis there)

`DashboardConfig.save()` calls `os.makedirs` and `json.dump` with no `try/except`. If the config directory (`/etc/broodforge/` by default) is not writable as the dashboard user (the systemd unit runs as `root`, which should be fine, but a misconfigured deployment may not be), `save()` raises and the auto-generated token is lost. The dashboard starts up with an empty `action_token` on the next restart, making all POST endpoints unauthenticated.

### F-033 · LOW · `forge_scripts.py phase-08` commits with `--allow-empty` which can silently create empty git commits

1. **F-001, F-002, F-017, F-028** — The forge completion signal is misleading. Either the phase scripts should fail explicitly when their work cannot be done, or forge.sh should emit a clear "FORGE INCOMPLETE — manual steps required for phases 04-05" banner rather than the "hatchery is operational" message.
