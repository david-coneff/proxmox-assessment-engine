Cloud Storage Setup

Backup destination credentials for restic + rclone

Created: 2026-05-31 · Review by: 2027-05-31 (cloud UIs change — verify steps are current)

Broodforge requires API-level access — not username + password — for all cloud backup destinations. Each provider section covers creating a credential scoped to a single bucket with minimum required permissions.

Backblaze B2 recommended

Backblaze B2 — straightforward setup, competitive pricing, native restic support

Authentication: Application Key (Key ID + Application Key string) · Restic: native B2 support, no rclone needed · Security: always use a bucket-scoped restricted key, never the Master Application Key

Step 1 — Create a Backblaze account
  1. Go to backblaze.com → Sign Up
  2. Verify your email address
Step 2 — Create a bucket
  1. Log in → B2 Cloud Storage → Buckets → Create a Bucket
  2. Bucket name: e.g. broodforge-cell-a-backup (must be globally unique)
  3. Files in bucket: Private
  4. Server-side encryption: Enable (SSE-B2 is free — adds a second layer on top of restic's own encryption)
Step 3 — Create a restricted Application Key
  1. Account → App Keys → Add a New Application Key
  2. Name: broodforge-backup
  3. Allow access to bucket: select the specific bucket (NOT "All Buckets")
  4. Type of access: Read and Write
  5. Allow List All Bucket Names: No
  6. Click Create New Key
  7. Copy both the Key ID and the Application Key immediately — Application Key is shown only once
If you lose the Application Key, delete this key and create a new one. There is no way to retrieve it after the creation screen.
Step 4 — Configure in broodforge (setup-backup.py) 
Provider: Backblaze B2
Key ID:   (paste the Key ID from step 3)
App Key:  (paste the Application Key from step 3)
Bucket:   broodforge-cell-a-backup
Path:     restic

Broodforge initialises a restic repository at b2:broodforge-cell-a-backup/restic and tests the connection before saving.

AWS S3

AWS S3 — dedicated IAM user with bucket-scoped policy

Authentication: IAM Access Key ID + Secret Access Key · Security: never use root credentials; create a dedicated IAM user with a minimal inline policy

Step 1 — Create an S3 bucket
  1. AWS Console → S3 → Create bucket
  2. Bucket name: e.g. broodforge-cell-a-backup (globally unique, lowercase, hyphens only)
  3. Block all public access: enabled (all four checkboxes)
  4. Server-side encryption: Enable — SSE-S3 (free)
Step 2 — Create an IAM user with a scoped policy
  1. AWS Console → IAM → Users → Create user
  2. Username: broodforge-backup
  3. Access type: Programmatic access only (no AWS Console login)
  4. Attach this inline policy:
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:ListBucket", "s3:GetBucketLocation"],
      "Resource": "arn:aws:s3:::broodforge-cell-a-backup"
    },
    {
      "Effect": "Allow",
      "Action": ["s3:GetObject", "s3:PutObject", "s3:DeleteObject"],
      "Resource": "arn:aws:s3:::broodforge-cell-a-backup/*"
    }
  ]
}

On the final screen, copy the Access Key ID and Secret Access Key — the Secret is shown only once.

Step 3 — Configure in broodforge 
Provider:        AWS S3
Region:          (your bucket's region, e.g. ap-southeast-2)
Access Key ID:   (paste from step 2)
Secret Key:      (paste from step 2)
Bucket:          broodforge-cell-a-backup
Path:            restic

Cloudflare R2

Cloudflare R2 — S3-compatible, no egress fees

Authentication: R2 API Token (separate from Cloudflare account API key) · Cost: 10 GB free/month, then ~$0.015/GB as of 2026 · Note: requires credit card on file even for free tier

Step 1 — Enable R2 and create a bucket
  1. dash.cloudflare.com → R2 → Get Started (enter credit card if prompted)
  2. R2 → Create bucket
  3. Bucket name: e.g. broodforge-cell-a-backup
Step 2 — Create a scoped R2 API Token
  1. R2 → Manage R2 API Tokens → Create API Token
  2. Token name: broodforge-backup
  3. Permissions: Object Read & Write
  4. Specify bucket: select the specific bucket (not all buckets)
  5. Create token — copy Access Key ID, Secret Access Key
  6. Note your Account ID (visible in the R2 dashboard right sidebar)

Your S3 endpoint: https://{Account ID}.r2.cloudflarestorage.com

Step 3 — Configure in broodforge 
Provider:        Cloudflare R2 (S3-compatible)
Endpoint:        https://{your-account-id}.r2.cloudflarestorage.com
Access Key ID:   (paste from step 2)
Secret Key:      (paste from step 2)
Bucket:          broodforge-cell-a-backup
Path:            restic

Broodforge configures rclone with type = s3, provider = Cloudflare, and the endpoint URL.

Google Drive

Google Drive — OAuth2 or Service Account (most complex setup)
Google Cloud Console UI changes frequently. If a step shows a different screen than described, check Google's current documentation. The overall flow remains the same.
OptionBest forComplexity
OAuth2 (Option A)Personal Google Drive; interactive setup onceMedium
Service Account (Option B)Shared Drive / Google Workspace; non-interactiveHigher
Option A — OAuth2 (personal Google Drive)

A.1 — Create a Google Cloud project

  1. console.cloud.google.com → project selector → New Project
  2. Name: broodforge-backup → Create

A.2 — Enable Google Drive API

  1. APIs & Services → Library → search Google Drive API → Enable

A.3 — Configure OAuth consent screen

  1. APIs & Services → OAuth consent screen
  2. User type: External
  3. App name: broodforge-backup, support/developer email: your email
  4. Save and Continue through all screens; add your Google account as a test user

A.4 — Create OAuth 2.0 credentials

  1. APIs & Services → Credentials → Create Credentials → OAuth client ID
  2. Application type: Desktop app · Name: broodforge-rclone
  3. Download the JSON file (or copy Client ID and Client Secret)

A.5 — Authorise rclone (run on your workstation, not the Proxmox host)

rclone config
# n) New remote → name> gdrive → Storage> drive
# client_id>   (paste Client ID)
# client_secret>   (paste Client Secret)
# scope> drive
# Use auto config? y   ← opens browser for OAuth authorisation

Copy ~/.config/rclone/rclone.conf to the Proxmox host at the same path.

A.6 — Configure in broodforge

Provider:      Google Drive (rclone remote)
rclone remote: gdrive
Path:          broodforge-backup/restic
Option B — Service Account (Shared Drive / Google Workspace)

A service account is a Google identity for applications — no human OAuth flow, no browser required. Better for production or Google Workspace environments.

B.1 — Create project and enable Drive API (same as A.1 and A.2)

B.2 — Create a Service Account

  1. IAM & Admin → Service Accounts → Create Service Account
  2. Name: broodforge-backup · Skip role assignment → Done

B.3 — Create and download a JSON key

  1. Click your service account → Keys → Add Key → Create new key → JSON
  2. Store the downloaded .json file in KeePass (treat as a private key)

B.4 — Share a Drive folder with the service account

  1. In Google Drive, create folder: broodforge-backup
  2. Right-click → Share → add the service account email → Role: Editor

B.5 — Configure rclone

rclone config
# n) New remote → name> gdrive-sa → Storage> drive
# client_id>   (leave blank)
# client_secret>   (leave blank)
# scope> drive
# service_account_file> /path/to/service-account-key.json
# Use auto config? n

B.6 — Configure in broodforge

Provider:      Google Drive (rclone remote — service account)
rclone remote: gdrive-sa
Path:          broodforge-backup/restic

Self-Hosted MinIO

Self-Hosted MinIO — S3-compatible, zero cloud dependency

Run on a spare node, NAS, or separate machine. Preferred over running on the Proxmox host itself for independence during recovery scenarios.

Step 1 — Install MinIO 
# On the MinIO host (Debian/Ubuntu)
wget https://dl.min.io/server/minio/release/linux-amd64/minio -O /usr/local/bin/minio
chmod +x /usr/local/bin/minio
mkdir -p /opt/minio/data
useradd -r -s /sbin/nologin minio

Create /etc/systemd/system/minio.service:

[Unit]
Description=MinIO Object Storage
After=network.target

[Service]
User=minio
EnvironmentFile=/etc/minio/config
ExecStart=/usr/local/bin/minio server /opt/minio/data --console-address :9001
Restart=always

[Install]
WantedBy=multi-user.target

Create /etc/minio/config (chmod 600):

MINIO_ROOT_USER=admin
MINIO_ROOT_PASSWORD=Change-Me-On-First-Run-1
systemctl enable --now minio
Store the MinIO root password in KeePass immediately after first login.
Step 2 — Create a bucket and access key via the console
  1. Open http://{minio-host}:9001 → log in → change the root password immediately
  2. Buckets → Create Bucket · Name: broodforge-backup
  3. Access Keys → Create Access Key · Name: broodforge-backup
  4. Copy Access Key and Secret Key
Step 3 — Configure in broodforge 
Provider:        MinIO (S3-compatible)
Endpoint:        http://{minio-host}:9000  (use https if TLS is configured)
Access Key ID:   (from step 2)
Secret Key:      (from step 2)
Bucket:          broodforge-backup
Path:            restic

Broodforge configures rclone with type = s3, provider = Minio, and the endpoint URL.
