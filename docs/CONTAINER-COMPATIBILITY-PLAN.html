2. Core Principle: Infrastructure vs Application

Compatibility checker boundary
✓ Infrastructure layer (this system checks)✗ Application layer (NOT automated — runbook documents these)
Image digest pullable from registryData inside volumes compatible with restored image version
Volume host paths exist on new systemApplication-internal config files referencing old IPs
Resource requirements met (RAM, CPU)Database schema migration requirements
Network bridges exist and are routableApplication state consistency after partial restore
All secret references still in registry
All declared hostname env vars resolvable via DNS Registry
Startup dependencies reachable in correct order
When infrastructure checks pass but application-level issues remain, the score is NEEDS_REVIEW with specific flags, not BLOCKED. The operator consults the per-service runbook for application-level steps.

4. Data Volume Backup Configuration

Backup methods (per container)
MethodBest forRecovery
PBS (Proxmox Backup Server)Full VM coverage; large disks; incremental + deduplicatedRestore VM from PBS, then start containers
Rclone volume backupGranular per-volume; high-frequency; small volumesRestore individual volumes, then start containers
Encrypted archiveSmall volumes; same as general external backup flowDownload, decrypt, extract, then start containers
Volume backup filename convention 
# Config state archive (existing pattern):
{cell_id}_{YYYY-MM-DD_HH_MM_SS}_{hash}.tar.gz.gpg

# Container volume archive (new):
{cell_id}_{vmname}_{container_name}_{volume_name}_{YYYY-MM-DD_HH_MM_SS}_{hash}.tar.gz.gpg

# Examples:
proxmox-cell-a_forgejo_postgresql_data_2026-05-31_02_00_00_a3f7b2.tar.gz.gpg
proxmox-cell-a_forgejo_forgejo-app_repos_2026-05-31_02_05_00_b4c8d1.tar.gz.gpg

# PBS offsite sync (managed by rclone):
{cell_id}_pbs_vm{vmid}_{vmname}_{YYYY-MM-DD_HH_MM_SS}_{hash}.tar.zst

{volume_name} is the declared volume name, not the full host path (paths contain slashes unsuitable for filenames).

5. Compatibility Check Algorithm

Seven-step check sequence (per container)
#CheckOn failure
1Image check — is digest pullable from registry?Registry unreachable → NEEDS_REVIEW · Digest missing → INCOMPATIBLE
2Volume path check — do host paths exist on restored system?Path missing + backup exists → NEEDS_REVIEW · No backup → INCOMPATIBLE
3Resource check — RAM, CPU, disk ≥ declared minimums?Any fail → INCOMPATIBLE
4Network check — bridge exists? hostname resolves?Bridge missing → INCOMPATIBLE · IP changed → NEEDS_REVIEW
5Secret check — all secret_references in Secret Registry?Missing → INCOMPATIBLE
6Env var check — any env var contains stale IP/hostname?Stale reference → NEEDS_REVIEW
7Dependency check — are startup_after containers compatible?Any dependency INCOMPATIBLE → BLOCKED

Final score = worst across all checks.

6. Score Meanings

ScoreMeaningAction
COMPATIBLEAll infrastructure checks passStart the container
NEEDS_REVIEWInfrastructure looks OK but something changed; container may workReview flags, then start. Check application-level issues per runbook.
INCOMPATIBLEAt least one infrastructure check failed; container will not startResolve flagged issues first
BLOCKEDA dependency is INCOMPATIBLEFix dependency chain from bottom up

Image digest pinning (Tier 2 assessment)

When containers are first deployed, the Tier 2 assessment:

  1. Queries running containers for their image digests
  2. Records the digest in bootstrap-state.json alongside the tag
  3. Future assessments detect if the digest has changed (image rebuilt or tag moved)

Collected by Tier 2 assessment via podman inspect / docker inspect. Key fields: cell_id, vmid, vm_name, container_runtime, containers[] (observed state: running/stopped, actual image digest, port bindings, volume mounts, resource usage), compose_files[] (detected files with current hashes).

10. Design Decisions

D-001: Digest pinning is required, tags are informational

Tags can be reassigned. Only the image digest provides a stable reference to an exact image. The system records digests at deployment time and flags drift.

D-002: Application-level compatibility is out of scope for automation

Database schema migration, application config file updates, and internal state consistency after restore are application-specific. The system flags that review is needed but does not attempt automated resolution. Per-service runbooks document the manual steps.

D-003: PBS is preferred for VMs with large disks

PBS is more efficient (incremental, compressed, deduplicated) than per-volume archives for large data sets. Per-volume rclone/encrypted-archive backup is better for frequently-changed small volumes where fine-grained recovery is needed.

D-004: Compose files live in Forgejo, not in bootstrap-state.json

Only the reference (repo URL, path, commit hash, file digest) is stored in bootstrap-state.json. The actual file is in Forgejo, making it version-controlled and diff-able.

D-005: Container minimum_capability feeds cell readiness

If the restored hardware cannot meet the declared minimum_capability of any container, that container scores INCOMPATIBLE, which propagates to the cell's readiness score. Capacity issues surface in the recovery readiness report before recovery is attempted.
