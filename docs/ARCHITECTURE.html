1. Foundational Philosophy

The platform's primary product is not virtual machines.
Applications (Forgejo, Nextcloud, monitoring) are workloads that run on the platform and benefit from its capabilities. They are not the reason the platform exists.
What the platform exists to produce

Reproducibility

Any component can be rebuilt from repository state. No manual knowledge required.

Recoverability

Any failure mode has a documented, tested recovery path — generated before it's ever needed.

Documentation

Everything that exists has a documented reason for existing, kept current automatically.

Operational intelligence

The platform continuously knows its own state and scores its own health against declared intent.

Concrete architectural consequences

  • The documentation engine and assessment engine are deployed before any user application workload
  • A workload that cannot be documented or assessed is not fit for deployment on this platform
  • Recovery capability is a deployment prerequisite, not a post-deployment concern
  • The platform continuously earns the right to host workloads by maintaining its own understanding of itself

2. Four Intelligence Phases

Each phase has a distinct runtime, purpose, and set of produced artifacts. Phases B/C/D are k3s workloads; Phase A is a pre-k3s stdlib Python tool that runs on the operator's workstation or the hatchery itself.

Phase A — Bootstrap Intelligence  (pre-k3s · stdlib Python · operator machine or hatchery)

Purpose: Build the first-ever environment from bare metal; spawn additional nodes onto an existing cluster.

Two processes:

ProcessProducesWhen
Forging ProcessForge package → bare hardware becomes first hatcheryFirst run only
Hatchery ProcessSpawn packages → broodlings join existing clusterSubsequent runs

Constraint: stdlib Python only — no pip, no external dependencies. Runs without network access from the forge package. All data from embedded manifests.

Phase B — Operational Intelligence  (k3s workload · runs continuously)

Purpose: Understand and document a running environment. Score its health continuously.

Produces
Living inventory — all VMs, services, volumes, network topology
Architecture documentation — what exists and why
Dependency maps — five graph types (service, network, data, secret, compute)
Drift reports — detected divergence between declared intent and observed reality
Five assessment scores — ACS, RRS, DCS, CRS, OSS (see Section 9)
Phase C — Recovery Intelligence  (k3s workload · Stargate Process · triggered by Phase B)

Purpose: Generate phoenix packages — reconstituting a failed node's identity on new hardware. Identity is preserved; underlying hardware adapts.

Produces
Phoenix packages — self-contained, offline-capable recovery archives
Readiness scores per node — is this node's recovery package current and actionable?
Recovery runbooks — step-by-step human-readable reconstruction guides
Recovery workbooks — HTML execution records for post-recovery audit
Phase D — Execution Intelligence  (k3s workload for generation · standalone for execution)

Purpose: Execute phoenix package scripts; capture failure packages for the improvement loop; learn from failures.

Produces
Phoenix scripts — identity-preserving bash with hardware adaptation at runtime
HTML workbooks — operator-fillable execution records
Failure packages — structured archives when a phase fails; sent to hatchery for analysis
Improvement loop — failure packages → LLM analysis → bootstrap repository updates

4. Source of Truth Hierarchy

AUTHORITATIVE — edit these directly 1. Metadata YAML files infrastructure intent (metadata/) 2. Git repositories declared state (OpenTofu, k8s manifests, configs) 3. OpenTofu state provisioned infrastructure 4. k3s / k8s API live running state DERIVED — never edit directly; regenerate from 1–4 5. Generated documentation 6. Generated phoenix/spawn packages and scripts 7. HTML workbook execution records (exception: these are the audit trail — operator fills them) DISPOSABLE — safe to delete and regenerate 8. Cached manifests, rendered templates, intermediate artifacts
Governance rules
  • Metadata YAML files are never generated — they are always set by the operator and reflect intent
  • When two sources disagree, the higher-ranked source wins; the assessment engine flags the discrepancy
  • Any change to sources 1–4 triggers a reassessment cycle
  • Generated artifacts (5–8) are reproducible: same inputs → same outputs every time
  • The only exception to "never edit derived" is the HTML workbook execution record — operators fill in steps as they complete them

8. Node Spawning — Hatchery Process

Overview: how a broodling is spawned
  1. Assessment engine reads hatchery's current cluster state — all reserved VMIDs, IPs, hostnames, k3s join tokens, cluster topology
  2. Operator selects execution mode first (Interactive vs Autonomous), then service disposition (full mirror / by group / individually)
  3. Conflict validator checks all proposed allocations — RED on any VMID/IP/hostname collision blocks package generation
  4. Spawn package assembled — self-contained archive copied to broodling after bare Proxmox install
  5. Broodling runs spawn.sh — executes phases 00–06 unattended (autonomous) or with a service menu (interactive)
  6. Bootstrap-state.json updated on hatchery and committed to Forgejo
Execution modes
ModeService selectionExecution
Autonomous (default)Selected on hatchery at package-generation time; locked into packagespawn.sh runs end-to-end unattended after KeePass unlock gate
InteractiveNo selection at generation time; all service scripts included in packagespawn.sh presents service menu on the broodling at runtime, evaluating actual hardware there
Asking execution mode first means Interactive operators skip the service selection flow entirely on the hatchery.

10. Architecture Decision Log

Decisions are numbered AD-NNN. The rationale for each decision is recorded here to prevent re-litigating closed questions.

AD-022 — k3s as primary application platform (over Podman)
Scheduling, failover, HA, standard workload definition format, rich API surface. k3s bootstraps into a fresh cluster via a single binary and joins the Proxmox cluster — compatible with the recovery model. Podman lacks the scheduling and HA story needed for multi-node operation.
AD-023 — Flux CD as GitOps engine (over ArgoCD)
flux bootstrap on a fresh cluster, pointed at Forgejo, reconciles the entire platform automatically. This recovery-bootstrapping advantage is decisive. ArgoCD requires a running cluster to install ArgoCD — a circular dependency during recovery.
AD-024 — Forgejo as sole Git provider; external providers are mirrors only
Forgejo runs as a pre-k3s VM — available even when k3s is down. Making it the source of truth means recovery can proceed without internet access. GitHub/GitLab are optional push mirrors for off-site backup only.
AD-025 — HTML as machine-generated workbook format
Self-contained HTML with embedded CSS/JS: no dependencies, browser-viewable without LibreOffice, print-friendly, machine-writable from Python stdlib. Checkboxes and collapsible sections are interactive; state persists in localStorage. Workbooks open directly from a package archive with no application install required.
AD-026 — Metadata YAML as primary source of infrastructure intent
Infrastructure intent (what should exist) lives in YAML files that are never generated. Everything else is derived from them. This makes intent explicit, version-controlled, and diff-able.
AD-027 — Four intelligence phases with distinct runtimes (pre/post k3s)
Phase A must run before k3s exists — stdlib Python only, no container runtime required. Phases B/C/D benefit from the k3s scheduler, persistent storage, and networking. Mixing the runtimes would make Phase A depend on what Phase A produces.
AD-028 — Documentation engine is the first k3s workload
Applications benefit from broodforge's documentation and recovery capabilities only if those capabilities are operational first. The documentation gate prevents deploying applications that cannot be documented or recovered.
AD-029 — Phoenix packages are self-contained and offline-capable; identity preserved, hardware adapts
A phoenix package must execute Phases 1–3 of recovery without network access. The node being recovered may have destroyed the network infrastructure it depends on. Identity preservation means the recovered node resumes its role with the same VMID, hostname, and IP — no cluster updates needed.
AD-030 — Failure packages structured for LLM analysis
The self-improvement loop: failure → structured failure package → LLM analysis → bootstrap repository update. Failure packages include the spawn manifest, hardware profile, logs, and checkpoints so an LLM has full context for diagnosis and fix suggestion.
AD-039 — Spawn packages are self-contained; broodling never queries hatchery API during execution
The spawn manifest embedded in the package is the point-in-time reservation snapshot. A WAN broodling may lose connectivity mid-execution. The package must be sufficient on its own. All VMID/IP/hostname conflicts are pre-checked at package generation time.
AD-040 — Initial deployment (forging) is an 8-phase process producing an operational hatchery
Phases: discover → plan → validate → host-config → VMs → k3s → GitOps → intelligence. Minimum viable forge deploys only the broodforge stack (doc engine + assessment engine). User applications are added via GitOps after intelligence gates pass.
AD-041 — Execution mode is asked first; Interactive operators skip service selection entirely
Asking mode (Interactive vs Autonomous) before service selection means Interactive operators never see the service selection flow — they answer one question and the package is built. Autonomous operators then proceed through service selection. This reduces friction for the common Interactive homelab case.
AD-042 — KeePass database is optionally embedded in packages; embedding doesn't compromise security
The database cannot be auto-unlocked — the operator must enter the master password at the KeePass gate before any secret is accessed. Embedding the database in the package means one fewer manual step during recovery. If not embedded, the operator provides a path (USB, local disk) at runtime. No secret value is ever written to the node filesystem in plain text.
AD-043 — Readable passphrase format: Capital.word.phrase.9
Leading capital, lowercase words, period separators, trailing single digit, 20–30 characters. Human-memorable, typeable in a terminal without shell escaping, satisfies most service requirements. Services that reject periods auto-detect failure and offer a plain alphanumeric regeneration. Service-specific format restrictions are recorded in service-catalog.yaml and applied automatically in future deployments.
AD-044 — Backup engine: restic + rclone, per-backup rotating restic password, three layers
Three layers: (1) KeePass DB — plain rclone copy, no restic (already AES-256; restic would create circular dependency with KeePass holding restic passwords); (2) configuration state — encrypted via restic; (3) application data volumes — opt-in, encrypted via restic. Each backup run rotates the restic repo password via restic key add + key remove; new password stored in KeePass at a timestamped path; secrets broker looks up the exact key per snapshot at restore time.
AD-045 — Timezone preference in bootstrap-state.json; all timestamps show UTC + local
Operator sets timezone at forge time; stored as vm_defaults.timezone. Documentation engine uses it for all generated output. All timestamps display both UTC and local time to remain unambiguous across readers in different timezones.
AD-046 — Proxmox nag suppression via JS patching with dpkg post-invoke hook
Proxmox Community Edition is fully functional without a subscription. The subscription popup is a commercial UI interruption. The patch targets a single condition in proxmoxlib.js. The dpkg hook re-applies the patch after proxmox-widget-toolkit or pve-manager package upgrades.
AD-047 — Headscale as self-hosted Tailscale coordination; hatchery is both spawn controller and tailnet coordinator
WAN spawn requires a secure channel between hatchery and broodling before SSH keys are exchanged. Headscale provides this via WireGuard tunnel without exposing SSH to the public internet. LAN spawns bypass Tailscale entirely. Broodlings remain permanent tailnet members after spawn.
AD-048 — Split-horizon DNS + two supported DNS/TLS provider paths
dnsmasq on the hatchery for LAN resolution; Cloudflare or DuckDNS for WAN/DDNS. host_identity.fqdn is the single canonical address — spawn packages embed the FQDN and work from LAN and WAN without modification. Cloudflare: certbot + cert-manager ClusterIssuer. DuckDNS: acme.sh + cert sync script (cert-manager has no native DuckDNS DNS-01 support). Squarespace has no DNS API — delegate to Cloudflare. inadyn archived Oct 2025 — not used.
AD-049 — Guided Setup Framework: four configuration modes
Autonomous (default): all settings auto-calculated from discovery. IP-Selective: fully autonomous except operator chooses IP addressing. Group-Manual: operator selects which setting groups to configure manually. Full Manual: operator walks through all settings with auto-suggestions. At every manual prompt: suggestion shown prominently; operator may accept or override; subsequent suggestions revised to be logically consistent with any override; conflicts warned but operator may proceed.
AD-050 — Two network deployment profiles: LAN-only (A) and WAN-capable (B)
LAN-only: flat network, dnsmasq for local resolution only, no Headscale, no external domain, self-signed TLS or none. WAN-capable: split-horizon DNS, Headscale for cross-network spawn, DDNS agent, Let's Encrypt DNS-01. Profile stored in network_topology.profile. Autonomous or guided migration between profiles via setup_network.py --migrate-to [lan|wan]. LAN→WAN: 7 steps (6 autonomous, 1 manual: router port forwarding). WAN→LAN: 4 steps (all autonomous).
