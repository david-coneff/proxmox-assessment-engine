DuckDNS Setup

Free DDNS and Let's Encrypt TLS for the broodforge hatchery

Created: 2026-05-31 · Review by: 2027-05-31 · See also: Cloudflare Setup (recommended if you have your own domain)

Trade-off vs Cloudflare: DuckDNS is free and simple, but your hatchery address will be myhatchery.duckdns.org rather than a subdomain of your own domain. cert-manager does not natively support DuckDNS — certs are synced via a shell script. If you have your own domain, the Cloudflare path is recommended.

Part 1 — DuckDNS DDNS Setup

Part 1 — DuckDNS Account and DDNS
Step 1.1 — Create a DuckDNS account and subdomain
  1. Go to duckdns.org
  2. Sign in with GitHub, Google, Twitter, or Reddit
  3. In the sub domain field, enter your chosen name (e.g. myhatchery) → add domain
  4. Your subdomain is now myhatchery.duckdns.org
  5. Copy your token — shown on the main dashboard (a UUID)
Store your token in KeePass at: Infrastructure/duckdns/token
Step 1.2 — Test the DDNS update

DuckDNS updates via a single HTTPS GET — no client software required.

# Update to your current WAN IP (ip= blank = auto-detect)
curl -s "https://www.duckdns.org/update?domains=myhatchery&token=YOUR-TOKEN&ip="

Expected response: OK
If you see KO, the token is wrong or the subdomain doesn't exist in your account.

Step 1.3 — Configure via broodforge (or manually)

During forging, forge.sh runs setup_ddns.py automatically. For manual setup:

python3 proxmox-bootstrap/setup_ddns.py \
  --state proxmox-bootstrap/bootstrap-state.json

Select DuckDNS, enter your subdomain name and KeePass path for the token.

Step 1.4 — Monitor the auto-update timer

Broodforge installs a systemd timer that updates DuckDNS every 5 minutes when the WAN IP changes.

# Check timer status
systemctl status broodforge-ddns.timer

# Check recent update runs
journalctl -u broodforge-ddns.service -n 20

# Force an immediate update
systemctl start broodforge-ddns.service

Part 2 — Let's Encrypt TLS Certificates

Broodforge uses acme.sh for Let's Encrypt when DuckDNS is selected. acme.sh has first-class DuckDNS DNS-01 support and no Python dependencies.

LayerToolCert covers
Proxmox host (Headscale, nginx)acme.shmyhatchery.duckdns.org, *.myhatchery.duckdns.org
k3s workloadsCert synced as k8s TLS secretSame wildcard cert — no cert-manager needed
Part 2a — Host certificate via acme.sh
Install acme.sh 
curl https://get.acme.sh | sh -s email=admin@myhatchery.duckdns.org
source ~/.bashrc

acme.sh installs to ~/.acme.sh/ and sets up a cron job for auto-renewal.

Issue the certificate 
export DuckDNS_Token="YOUR-DUCKDNS-TOKEN"

~/.acme.sh/acme.sh --issue \
  --dns dns_duckdns \
  -d myhatchery.duckdns.org \
  -d "*.myhatchery.duckdns.org" \
  --server letsencrypt \
  --keylength 2048 \
  --dnssleep 60
--dnssleep 60 gives DuckDNS time to propagate the TXT record before Let's Encrypt checks it. If the challenge fails, increase to 120.
Install cert to a stable path 
mkdir -p /etc/broodforge/ssl

~/.acme.sh/acme.sh --install-cert \
  -d myhatchery.duckdns.org \
  --cert-file      /etc/broodforge/ssl/cert.pem \
  --key-file       /etc/broodforge/ssl/key.pem \
  --fullchain-file /etc/broodforge/ssl/fullchain.pem \
  --reloadcmd "systemctl reload headscale nginx 2>/dev/null || systemctl restart headscale"

After every successful renewal, acme.sh copies the cert files and runs the reload command.

Configure Headscale

In /etc/headscale/config.yaml:

tls_cert_path: /etc/broodforge/ssl/fullchain.pem
tls_key_path: /etc/broodforge/ssl/key.pem
systemctl restart headscale
Verify auto-renewal 
# Test without actually renewing
~/.acme.sh/acme.sh --renew -d myhatchery.duckdns.org --force --staging

# Check cron entry
crontab -l | grep acme

acme.sh renews automatically when within 30 days of expiry.

Part 2b — k3s workload certificates (cert sync)

cert-manager has no native DuckDNS DNS-01 support. Broodforge syncs the wildcard cert issued by acme.sh into a Kubernetes TLS secret and refreshes it on renewal.

Create the Kubernetes TLS secret 
kubectl create secret tls duckdns-wildcard-tls \
  --namespace default \
  --cert=/etc/broodforge/ssl/fullchain.pem \
  --key=/etc/broodforge/ssl/key.pem

For other namespaces (monitoring, applications):

kubectl get secret duckdns-wildcard-tls -o yaml \
  | sed 's/namespace: default/namespace: monitoring/' \
  | kubectl apply -f -
Auto-sync the secret on cert renewal

Create /opt/broodforge/sync-cert-to-k8s.sh:

#!/bin/bash
# Sync renewed DuckDNS wildcard cert into Kubernetes TLS secrets
set -e
for NS in default monitoring applications; do
  kubectl create secret tls duckdns-wildcard-tls \
    --namespace "$NS" \
    --cert=/etc/broodforge/ssl/fullchain.pem \
    --key=/etc/broodforge/ssl/key.pem \
    --dry-run=client -o yaml | kubectl apply -f -
done
echo "[cert-sync] Wildcard cert synced to namespaces: default monitoring applications"
chmod +x /opt/broodforge/sync-cert-to-k8s.sh

Wire the sync script to acme.sh renewal:

~/.acme.sh/acme.sh --install-cert \
  -d myhatchery.duckdns.org \
  --cert-file      /etc/broodforge/ssl/cert.pem \
  --key-file       /etc/broodforge/ssl/key.pem \
  --fullchain-file /etc/broodforge/ssl/fullchain.pem \
  --reloadcmd "/opt/broodforge/sync-cert-to-k8s.sh"
Use the cert in Ingress resources 
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: forgejo
  namespace: default
spec:
  tls:
  - hosts:
    - forgejo.myhatchery.duckdns.org
    secretName: duckdns-wildcard-tls   # uses the synced cert
  rules:
  - host: forgejo.myhatchery.duckdns.org
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: forgejo
            port:
              number: 3000

When acme.sh renews the cert, sync-cert-to-k8s.sh updates the secret and Kubernetes serves the new cert on the next request — no Ingress restart needed.

  "ddns_zone":                  "duckdns.org",
  "ddns_record":                "myhatchery",
  "ddns_credential_reference":  "Infrastructure/duckdns/token",
  "ssl_provider":               "acme.sh",
  "ssl_method":                 "dns-01-duckdns",
  "ssl_cert_path":              "/etc/broodforge/ssl/fullchain.pem",
  "ssl_key_path":               "/etc/broodforge/ssl/key.pem"
}

The documentation engine uses these fields to pre-populate recovery runbook steps for certificate retrieval and renewal.

Forge-Time Summary

When forge.sh runs, you provide once, at forge time:

  1. DuckDNS subdomain name → stored in bootstrap-state.json host_identity.fqdn
  2. KeePass path for DuckDNS token → retrieved at each step automatically
  3. Email for Let's Encrypt account → stored in bootstrap-state.json

Everything else — acme.sh install, cert issuance, Headscale config, k8s secret, sync script, DDNS timer — is automated.

Limitations vs Cloudflare

FeatureDuckDNSCloudflare
Own domainNo — *.duckdns.org onlyYes — your domain
cert-manager native integrationNo — manual syncYes — ClusterIssuer
Wildcard certYes (via acme.sh)Yes (via certbot or acme.sh)
Additional DNS recordsNoYes
API reliabilityGoodExcellent
Migrating from DuckDNS → Cloudflare (when you acquire your own domain)
  1. Set up Cloudflare following CLOUDFLARE-SETUP.html
  2. Update host_identity.fqdn and network_topology.* in bootstrap-state.json
  3. Re-issue certs with certbot (or keep acme.sh with Cloudflare)
  4. Update the ClusterIssuer to Cloudflare DNS-01
  5. Update any external references to the old duckdns.org address
  6. Re-generate forge-time documentation: engine.py --mode bootstrap --manifest proxmox-bootstrap/bootstrap-state.json

Troubleshooting

acme.sh: DuckDNS DNS-01 challenge fails

Increase --dnssleep to 120 seconds. Check the TXT record was created:

dig TXT _acme-challenge.myhatchery.duckdns.org +short

If empty, the DuckDNS token may be wrong or the API call failed. Check: journalctl -u cron -n 20

k8s secret not updating after cert renewal 
journalctl -u cron -n 20 | grep acme
# Run manually:
/opt/broodforge/sync-cert-to-k8s.sh
Headscale shows expired cert 
~/.acme.sh/acme.sh --renew -d myhatchery.duckdns.org --force
systemctl restart headscale
DDNS update failing 
# Test directly — should return OK
curl -s "https://www.duckdns.org/update?domains=myhatchery&token=YOUR-TOKEN&ip="

Check that your token in KeePass matches the one on the DuckDNS dashboard.
