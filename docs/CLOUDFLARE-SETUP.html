Recommended DNS provider. Cloudflare has a well-supported API, a generous free tier, and integrates cleanly with certbot, cert-manager, and DNS-01 certificate challenges. If your domain is registered elsewhere (Squarespace, GoDaddy, Namecheap), you can delegate DNS to Cloudflare without moving your registration — Cloudflare handles DNS only.

What Cloudflare provides for broodforge

Part 1 — DNS Setup

Part 1 — DNS Setup (account · domain · records · API token)
Step 1.1 — Create a Cloudflare account
  1. Go to cloudflare.com → Sign Up
  2. Free plan is sufficient for all broodforge needs
  3. Verify your email
Step 1.2 — Add your domain to Cloudflare
  1. Dashboard → Add a site → enter your domain (e.g. home.example.com)
  2. Select the Free plan
  3. Cloudflare scans existing DNS records — keep any MX records (email), delete records you no longer need
  4. Cloudflare shows two nameserver addresses — copy both (needed for the next step)
Step 1.3 — Update nameservers at your registrar
Squarespace
  1. domains.squarespace.com → click your domain
  2. DNS → Nameservers → Use Custom Nameservers
  3. Delete the existing Squarespace nameservers; add both Cloudflare nameservers → Save

Propagation: minutes to 48 hours

GoDaddy
  1. GoDaddy → My Products → your domain → DNS
  2. Scroll to Nameservers → Change → Enter my own nameservers
  3. Enter both Cloudflare nameservers → Save
Namecheap
  1. Dashboard → your domain → Manage → Nameservers
  2. Select Custom DNS → enter both Cloudflare nameservers → Save

Verify propagation

# Should return Cloudflare nameservers (e.g. alice.ns.cloudflare.com)
dig NS home.example.com +short
Step 1.4 — Add DNS records in Cloudflare

Cloudflare dashboard → your domain → DNS → Records → Add record

TypeNameValueProxyNotes
Ahatcheryyour WAN IPDNS only (grey)Hatchery's public address
Aforgejoyour WAN IPDNS only (grey)If Forgejo should be externally accessible
Proxy must be DNS only (grey cloud). Headscale's gRPC and Forgejo's SSH/Git protocols are not compatible with Cloudflare's HTTP proxy. Orange cloud = connectivity broken.
Step 1.5 — Create a scoped API token required

Broodforge needs an API token with just enough permission to manage DNS records and create TXT records for Let's Encrypt challenges.

  1. Cloudflare dashboard → top right → My Profile → API Tokens
  2. Create Token → use the Edit zone DNS template
  3. Configure:
    • Token name: broodforge
    • Permissions: Zone / DNS / Edit (pre-filled)
    • Zone resources: Include / Specific zone / your domain
  4. Continue to summary → Create Token
  5. Copy the token immediately — shown only once
Store in KeePass at: Infrastructure/cloudflare/api-token
forge.sh retrieves it automatically — you never paste it manually again.

Part 2 — DDNS Auto-Update

Part 2 — DDNS Auto-Update (skip if WAN IP is static)

Broodforge installs a systemd timer (broodforge-ddns.timer) that checks your WAN IP every 5 minutes and updates the Cloudflare A record when it changes.

Step 2.1 — Install dns-lexicon 
pip install dns-lexicon
Step 2.2 — Test the update manually 
LEXICON_CLOUDFLARE_TOKEN="your-cloudflare-api-token" \
  lexicon cloudflare update home.example.com A \
  --name hatchery --content "$(curl -s ifconfig.me)"

Expected: no error output, Cloudflare A record updates to your current WAN IP.

Step 2.3 — Configure via broodforge (or manually)

During forging, forge.sh runs setup_ddns.py automatically. For manual setup:

python3 proxmox-bootstrap/setup_ddns.py \
  --state proxmox-bootstrap/bootstrap-state.json

Select Cloudflare, enter your domain zone, record name, and KeePass path for the API token.

Step 2.4 — Monitor the timer 
# Check timer status
systemctl status broodforge-ddns.timer

# Check recent update runs
journalctl -u broodforge-ddns.service -n 20

# Force an immediate update
systemctl start broodforge-ddns.service

Part 3 — TLS Certificates

DNS-01 challenges. All Let's Encrypt certificates use DNS-01 — port 80 never needs to be open. Wildcard certs (*.home.example.com) are supported. Works behind strict firewalls.
LayerToolCert covers
Proxmox host (Headscale, nginx)certbot or acme.shhatchery.home.example.com, *.home.example.com
k3s workloadscert-managerPer-service certs via ClusterIssuer

Two tools available for host-level certs — choose one (certbot below, acme.sh in 3a-alt). cert-manager (3b) is required regardless of which you choose.

Part 3a — Host certificate via certbot recommended
Install certbot + Cloudflare plugin 
apt install certbot python3-certbot-dns-cloudflare
Create the Cloudflare credentials file 
mkdir -p /etc/broodforge
cat > /etc/broodforge/cloudflare.ini << 'EOF'
# Cloudflare API token — KeePass: Infrastructure/cloudflare/api-token
dns_cloudflare_api_token = PASTE_TOKEN_HERE
EOF
chmod 600 /etc/broodforge/cloudflare.ini
During forging, forge.sh writes this file from KeePass automatically.
Obtain the wildcard certificate 
certbot certonly \
  --dns-cloudflare \
  --dns-cloudflare-credentials /etc/broodforge/cloudflare.ini \
  --dns-cloudflare-propagation-seconds 30 \
  -d "hatchery.home.example.com" \
  -d "*.home.example.com" \
  --email admin@home.example.com \
  --agree-tos \
  --non-interactive

Certificates land in /etc/letsencrypt/live/hatchery.home.example.com/.

Configure Headscale to use the certificate

In /etc/headscale/config.yaml:

tls_cert_path: /etc/letsencrypt/live/hatchery.home.example.com/fullchain.pem
tls_key_path: /etc/letsencrypt/live/hatchery.home.example.com/privkey.pem
systemctl restart headscale
Configure auto-renewal deploy hook

certbot installs a systemd timer for renewal. Add a hook to reload Headscale after renewal:

cat > /etc/letsencrypt/renewal-hooks/deploy/reload-headscale.sh << 'EOF'
#!/bin/bash
systemctl reload headscale || systemctl restart headscale
EOF
chmod +x /etc/letsencrypt/renewal-hooks/deploy/reload-headscale.sh

# Verify renewal works
certbot renew --dry-run
Part 3a-alt — Host certificate via acme.sh alternative

Shell-only alternative to certbot — no Python dependencies. Same result: wildcard cert at a stable path, auto-renewed. Use this if you prefer to avoid Python tooling on the host.

Install acme.sh 
curl https://get.acme.sh | sh -s email=admin@home.example.com
source ~/.bashrc
Issue the certificate 
export CF_Token="YOUR-CLOUDFLARE-API-TOKEN"

~/.acme.sh/acme.sh --issue \
  --dns dns_cf \
  -d "hatchery.home.example.com" \
  -d "*.home.example.com" \
  --server letsencrypt \
  --dnssleep 30
Install cert to a stable path 
mkdir -p /etc/broodforge/ssl

~/.acme.sh/acme.sh --install-cert \
  -d "hatchery.home.example.com" \
  --cert-file      /etc/broodforge/ssl/cert.pem \
  --key-file       /etc/broodforge/ssl/key.pem \
  --fullchain-file /etc/broodforge/ssl/fullchain.pem \
  --reloadcmd "systemctl reload headscale 2>/dev/null || systemctl restart headscale"
Using /etc/broodforge/ssl/ means the same Headscale config template works for both Cloudflare and DuckDNS paths.
Verify auto-renewal 
crontab -l | grep acme
~/.acme.sh/acme.sh --renew -d "hatchery.home.example.com" --force --staging
Part 3b — k3s workload certificates (cert-manager) required

cert-manager runs inside k3s and issues per-service TLS certificates automatically using the same Cloudflare API token.

Create the Cloudflare API token secret in k3s 
kubectl create namespace cert-manager

kubectl create secret generic cloudflare-api-token \
  --namespace cert-manager \
  --from-literal=api-token="$(grep api_token /etc/broodforge/cloudflare.ini | awk '{print $3}')"
forge.sh generates this secret from KeePass — no manual token pasting needed.
Create the ClusterIssuer 
# /etc/broodforge/k8s/clusterissuer-cloudflare.yaml
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: letsencrypt-prod
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory
    email: admin@home.example.com
    privateKeySecretRef:
      name: letsencrypt-prod-account-key
    solvers:
    - dns01:
        cloudflare:
          apiTokenSecretRef:
            name: cloudflare-api-token
            key: api-token
kubectl apply -f /etc/broodforge/k8s/clusterissuer-cloudflare.yaml
Request a certificate for a service (Ingress annotation) 
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: forgejo
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  tls:
  - hosts:
    - forgejo.home.example.com
    secretName: forgejo-tls
  rules:
  - host: forgejo.home.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: forgejo
            port:
              number: 3000

cert-manager sees the annotation, performs the DNS-01 challenge, and stores the cert in forgejo-tls. Renewal is fully automatic.

Verify cert-manager is working 
# Check ClusterIssuer status
kubectl describe clusterissuer letsencrypt-prod

# Check certificate status
kubectl describe certificate forgejo-tls -n default

# Watch the challenge process
kubectl get challenges -A -w

Forge-Time Setup Summary

When forge.sh runs, it handles all of the above automatically. You provide once, at forge time:

  1. Domain name → stored in bootstrap-state.json host_identity.domain
  2. KeePass path for Cloudflare API token → retrieved at each step automatically
  3. Email for Let's Encrypt account → stored in bootstrap-state.json
  4. Whether WAN IP is static or dynamic → configures DDNS timer if dynamic

Everything else — certbot install, credentials file, cert issuance, Headscale config, k8s secret, ClusterIssuer — is automated.

Troubleshooting

certbot: DNS propagation timeout

Increase --dns-cloudflare-propagation-seconds to 60 or 120. Cloudflare propagates quickly but Let's Encrypt occasionally polls before the change is visible.

cert-manager challenge stuck 
kubectl describe challenge -A

Check the Cloudflare API token has Zone / DNS / Edit permission for the correct zone.

Headscale shows TLS error after cert renewal 
systemctl status headscale
# If not running:
certbot renew --force-renewal
systemctl restart headscale
Cloudflare "This site can't be reached"

Check that the A record's proxy status is DNS only (grey), not proxied (orange). Headscale's gRPC protocol is not compatible with Cloudflare's HTTP proxy.

Nameserver propagation check 
dig NS home.example.com @8.8.8.8 +short
# Should show alice.ns.cloudflare.com, bob.ns.cloudflare.com (your assigned NS)
